title: Access Model
description: How users, agents, API keys, and share links are authorized.
---

## Principals

Every request is evaluated as a principal. A principal can be a signed-in user, an agent credential, an API key, or a scoped share link.

That shared model matters: web UI actions, REST calls, CLI commands, and MCP tools all ask the same policy layer before reading or mutating artifacts.

## Default Posture

Artifacts are deny-by-default unless a policy grants access. View and edit capabilities are modeled independently so a link can allow reading without granting mutation.

## Common Policies

- Private artifacts are visible only to authorized principals.
- Public artifacts can be viewed by anyone with the URL.
- Allowlisted access grants named users or emails explicit access.
- Share links grant scoped viewer or editor access and can be revoked.

## Billing Gates

Some collaboration features, such as share links, can be gated by account entitlements. When a feature is unavailable, the API and MCP tools fail before creating the privileged object.

## Audit Events

Sensitive changes emit audit events. Use audit history to review artifact changes, access changes, share-link lifecycle, and workspace activity.
